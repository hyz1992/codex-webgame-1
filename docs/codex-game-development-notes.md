# Codex 游戏开发使用心得

本文以《切线冲刺》为例，记录当前阶段使用 Codex 开发游戏的能力边界、最佳实践，以及本项目“几乎纯聊天式推进”的经验得失。

## 一句话结论

Codex 目前非常适合做浏览器游戏的工程型开发：搭项目、拆模块、写 TypeScript、接 Phaser、做调试面板、补测试、处理资源管线、根据截图和录屏迭代问题。它也能辅助做游戏设计和视觉方向，但还不能替代成熟游戏引擎编辑器、专业美术管线、动画师、关卡设计师和长期手感测试。

如果目标是浏览器游戏、移动端 H5、Web 端原型或 Phaser/Three.js/R3F 项目，Codex 的性价比很高。如果目标是 Unity、Godot、Cocos Creator 这类强编辑器工作流，Codex 仍能写脚本、分析逻辑、整理方案，但对编辑器内场景搭建、资源绑定、动画状态机和真实设备调试的掌控力会弱很多。

## Codex 现在擅长什么

Codex 在本项目中表现比较好的能力包括：

- 快速把想法落成可运行的浏览器项目。
- 在 Phaser 3、TypeScript、Vite、Vitest 之间建立稳定工程结构。
- 把输入、状态、碰撞、生成、视觉投影和 UI 分层。
- 按用户反馈连续修复 bug，例如重开、暂停、护盾、碰撞、调试模式等问题。
- 根据截图和录屏推断视觉穿帮点，并把问题转化为可调常量或投影算法。
- 写测试保护核心逻辑，例如碰撞、状态、投影、资源清单和输入映射。
- 处理透明 PNG、序列帧裁剪、资源清单和资产显示尺寸。
- 生成中文提交日志、阶段性文档和协作说明。

这类工作本质上是“工程、验证、调参、整理上下文”，Codex 很适合。

## Codex 现在不擅长什么

游戏开发里仍有几类工作不能完全交给 Codex：

- 高质量原创美术。Codex 可以写提示词、处理抠图、切图和资产入库，但最终画风、角色魅力和商业品质仍依赖稳定的美术管线。
- 精细动画。Codex 能接序列帧和调帧率，但动作的重心、节奏、姿态张力仍需要专业动画判断。
- 真实手感。移动端滑动、反应窗口、速度曲线、失败挫败感，都需要人反复试玩。
- 复杂 3D 引擎编辑器工作流。Unity、Godot、Cocos Creator 的场景、Prefab、动画机和 Inspector 绑定，Codex 没有浏览器项目那么顺手。
- 大规模关卡设计。Codex 可以生成规则和测试无解路线，但“好玩、有节奏、有记忆点”的关卡仍要人为把关。
- 长期性能调优。WebGL、移动端 GPU、纹理内存、包体和多机型适配，需要真实设备数据。

可以把 Codex 理解为非常强的工程搭档和迭代执行者，而不是完整替代制作人、美术、动画、关卡和 QA。

## 最佳实践步骤

使用 Codex 做游戏，建议按下面步骤推进：

1. 先确定平台和引擎
   如果目标是 H5 或浏览器游戏，优先 Phaser 3、Three.js 或 React Three Fiber。除非项目明确需要原生引擎能力，否则不要一开始就上 Unity/Godot/Cocos。

2. 先写核心循环
   用几句话固定玩家目标、输入方式、失败条件、奖励和单局节奏。不要一开始纠结完整世界观、商店和成长系统。

3. 做最低可玩版本
   用方块、线条和占位图先跑通开始、游玩、失败、结算、重开。核心循环不成立时，美术越精细越浪费。

4. 提前做 PC 调试入口
   移动端项目也要支持桌面键盘、鼠标手势、`?debug=1`、大量护盾、调试面板和快速重开。这样 Codex 和用户都能高频验证。

5. 拆清楚状态和渲染
   分离 `RunState`、输入、碰撞、生成、投影和 Phaser 场景。能写单元测试的逻辑不要埋在渲染循环里。

6. 用测试保护基础规则
   对输入映射、碰撞结算、护盾、加速、投影、资源清单、无解路线等写测试。视觉可以靠截图，规则必须靠测试。

7. 用截图和录屏反馈视觉问题
   对“看起来不自然”的问题，截图和录屏比文字更有效。Codex 可以从画面里定位透视、锚点、缩放、深度和运动节奏问题。

8. 在核心玩法稳定后再换资产
   程序化 UI 和占位图适合验证，正式风格需要统一资产规范：视角、透明背景、锚点、尺寸、色彩、轮廓、远处可读性。

9. 小步提交
   每次只提交一个主题，例如“修复暂停”“调整跑道透视”“替换摩托序列帧”。这样后续回看和回滚都清楚。

10. 保持文档更新
    `README.md` 记录如何运行和当前玩法，`AGENTS.md` 记录给代理的工作约束，设计文档记录为什么这么做。

## 本项目做得好的地方

本项目比较成功的做法包括：

- 很早确定了移动端优先，同时要求 PC 端可调试和预览。
- 没有一开始追求完整大作，而是先把短局跑酷核心循环跑通。
- 用户持续提供截图和录屏，让视觉问题有了具体证据。
- 多次把用户感受转化成工程问题，例如跑道尽头、障碍物漂移、路灯聚拢、车道线穿帮。
- 逐步把魔法数沉淀为 `layout.ts` 中的中文注释常量，方便用户自行微调。
- 使用测试保护核心逻辑，不只靠肉眼判断。
- 采用频繁中文提交，项目演化历史比较容易追踪。

尤其值得保留的是“用户看效果，Codex 改实现”的循环。游戏体验很难一次设计正确，短反馈链路比长计划更重要。

## 本项目做得不够好的地方

本项目也有一些明显可以改进的地方：

- `AGENTS.md` 和完整 `README.md` 出现得太晚，早期约束主要散落在聊天记录里。
- 初期设计里有跳跃和下滑，后来发现伪 3D 表现和操作体验不匹配才移除，说明核心输入可以更早收敛。
- 程序化 UI 和程序化障碍物探索时间偏长，正式资产风格规范建立得偏晚。
- 伪 3D 算法经历多轮试错后才稳定。如果一开始就把“物体固定在世界坐标，人向前移动，再投影到屏幕”作为统一模型，会更快。
- 资产生成缺少一份早期风格圣经，例如主色、材质、视角、轮廓粗细、透明边缘、帧尺寸和锚点规则。
- 浏览器自动截图验证链路不够稳定，很多视觉验收依赖用户截图。如果能稳定自动截图和像素检查，效率会更高。
- 临时预览和中间产物较多，需要更明确的资产源文件目录、导出目录和清理策略。

这些问题并不罕见。聊天式做项目的优点是快，缺点是早期决策容易被即时反馈牵着走，所以更需要轻量文档和验收清单。

## 尚未充分应用的提效做法

后续如果继续把项目推向发布版，可以加入这些做法：

- 建立视觉风格板：主角、道路、城市、障碍物、道具、HUD 都用同一套风格规则。
- 建立资产规格表：每类资产的尺寸、锚点、占道宽度、透明背景、命名、压缩方式和验收图。
- 建立固定录屏点：普通模式 30 秒、调试模式 2 分钟、左中右车道、道具拾取、撞击、暂停重开。
- 引入截图对比：至少固定几个视口，检查 HUD 不遮挡、跑道不断层、物件不漂移。
- 维护短 backlog：把“必须发布前解决”和“可以以后再做”分开。
- 为手感参数做调试 UI：速度、生成间隔、道路透视、主角大小、道具大小可以在面板里实时调。
- 更早使用专业资产流程：参考图、统一提示词、透明背景、批量切图、边缘检查、入库预览。
- 对移动端真机做周期性测试：浏览器桌面预览无法完全代表触摸延迟、屏幕尺寸和性能。

## 关于“纯聊天式做项目”

本项目很大程度上是通过自然语言对话推进的：用户描述目标、反馈截图和录屏，Codex 读代码、改实现、生成资源、跑测试、提交。这个模式已经能做出一个可玩的浏览器游戏原型，并持续向发布品质靠近。

但“纯聊天式”不等于“不需要工程方法”。真正让项目能走下去的，是聊天背后的这些结构：

- 每轮反馈尽量具体。
- 代码分层清楚。
- 调试入口随时可用。
- 测试覆盖核心规则。
- 常量可调。
- 提交颗粒度小。
- 文档记录当前事实。

换句话说，Codex 能把聊天转成工程动作，但项目仍然需要清晰目标、验证证据和持续取舍。把 Codex 当成会写代码、会查问题、会整理上下文的制作搭档，效果会比把它当成一次性“生成游戏”的工具好得多。

## 当前建议

接下来最值得做的不是继续堆功能，而是建立发布前体验清单：

- 主角动画是否清晰、有速度感。
- 障碍物是否一眼可读。
- 道具是否足够大、足够好认。
- 跑道透视是否稳定。
- 碰撞是否符合直觉。
- 失败反馈是否明确。
- 一局前 30 秒是否足够爽快。
- 移动端是否真的顺手。

这些问题解决后，再考虑商店、皮肤、排行榜、任务、音效和更多障碍组合，项目会更稳。
